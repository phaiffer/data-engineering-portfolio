# API Layer

The API exists to expose the modeled DuckDB marts through a small read-only JSON interface. It is designed as a local portfolio serving layer and a clean contract for future dashboard consumption.

The API does not perform analytical transformations. Business logic belongs in Python processing and DBT marts, not Flask routes.

## Runtime

- Framework: Flask.
- Database: local DuckDB file generated by DBT.
- Default URL: `http://127.0.0.1:5002`.
- Local protocol: HTTP only.
- Expected DuckDB path: `data/retail_revenue_analytics.duckdb`.

Do not use `https://127.0.0.1:5002` for local development. Flask is not serving TLS in this project.

`GET /` returning `404` is expected. The Flask process serves API routes only; use `/health` or `/api/v1` as the local entry points.

## Docker Runtime

The project also includes a Docker packaging path for the API:

- Dockerfile: `docker/api.Dockerfile`
- Compose service: `retail-api`
- Exposed host port: `5002`
- Mounted DuckDB path inside the container:
  `/workspace/projects/03-retail-revenue-analytics/data/retail_revenue_analytics.duckdb`

The API container does not bake generated DuckDB data into the image. It reads the bind-mounted `data/` directory at runtime.

If the mounted DuckDB file is missing, the API still starts, but `/health` returns a degraded status and mart endpoints cannot serve modeled results yet.

Docker is optional here. The API can also run directly from the local Python environment after the DuckDB marts exist.

## Local CORS

The API supports explicit CORS origin control through `RETAIL_REVENUE_API_CORS_ALLOWED_ORIGINS`.

The default local dashboard origins include:

- `http://127.0.0.1:4173`
- `http://localhost:4173`
- `http://127.0.0.1:5173`
- `http://localhost:5173`
- `http://127.0.0.1:5174`
- `http://localhost:5174`
- `http://127.0.0.1:5175`
- `http://localhost:5175`
- `http://127.0.0.1:5179`
- `http://localhost:5179`

When `RETAIL_REVENUE_API_ALLOW_LOCAL_DEV_CORS=true`, the API also allows HTTP `localhost` and `127.0.0.1` browser origins on local Vite-style ports from `5173` through `5199`. This makes local Vite port fallback ergonomic without using unrestricted wildcard CORS.

In the Docker Compose demo path, the API is configured to allow the containerized dashboard origin on port `4173` while still keeping the local Vite convenience rule enabled.

Direct browser navigation to `/health` does not require CORS. Dashboard `fetch` calls do. That means `/health` can open directly while the dashboard is still blocked if the Vite origin is not allowed.

## Data Sources

The API reads from DBT mart tables:

- `fct_sales`
- `mart_daily_revenue`
- `mart_category_performance`
- `mart_seller_performance`
- `mart_customer_state_performance`
- `mart_order_status_summary`
- `mart_payment_type_summary`

It does not read raw Bronze files or Silver CSV files directly.

## Endpoints

### GET /health

Returns API status, project name, API layer description, DuckDB path, and whether the database file exists.

### GET /api/v1

Returns a concise index of available endpoints and scope notes.

### GET /api/v1/kpis

Returns dashboard-friendly KPIs from `fct_sales`:

- `total_orders`
- `total_order_items`
- `total_item_revenue`
- `total_freight_value`
- `total_gross_merchandise_value`
- `average_order_value_item_side`

These are item-side analytical measures, not accounting-grade revenue.

### GET /api/v1/daily-revenue

Reads from `mart_daily_revenue`.

Query parameters:

- `limit`
- `order_status`
- `sort`, either `asc` or `desc`

### GET /api/v1/category-performance

Reads from `mart_category_performance`.

Query parameters:

- `limit`
- `sort_by`
- `sort`
- `min_item_revenue`

### GET /api/v1/seller-performance

Reads from `mart_seller_performance`.

Query parameters:

- `limit`
- `seller_state`
- `sort_by`
- `sort`

### GET /api/v1/customer-state-performance

Reads from `mart_customer_state_performance`.

Query parameters:

- `limit`
- `customer_state`
- `sort_by`
- `sort`

### GET /api/v1/order-status-summary

Reads from `mart_order_status_summary`.

### GET /api/v1/payment-type-summary

Reads from `mart_payment_type_summary`.

Payment summaries describe payment behavior. They are not item-level sales revenue.

### GET /api/v1/fct-sales

Returns a small exploratory sample from `fct_sales`.

Query parameters:

- `limit`, capped at 100
- `order_status`
- `customer_state`
- `seller_state`
- `category`
- `sort_by`
- `sort`

This endpoint is intentionally bounded and should not be treated as a bulk export API.

## Response Shapes

List endpoints return:

```json
{
  "data": [],
  "row_count": 0,
  "filters": {}
}
```

KPI endpoint returns:

```json
{
  "data": {},
  "notes": []
}
```

Errors use:

```json
{
  "error": "error_code",
  "message": "Human-readable message",
  "details": "Optional details"
}
```

## Non-Goals

- No write endpoints.
- No authentication or authorization.
- No caching.
- No background jobs.
- No DBT execution on request.
- No orchestration.
- No cloud deployment.
- No production API hardening claims.

## Troubleshooting

If `/health` reports a degraded status or mart-backed endpoints return `503`, the usual cause is a missing DuckDB mart file at `data/retail_revenue_analytics.duckdb`.

Wrong:

```text
https://127.0.0.1:5002
```

Right:

```text
http://127.0.0.1:5002
```

If the dashboard reports an unavailable API, check:

- the Flask process is running;
- `VITE_API_BASE_URL` points to `http://127.0.0.1:5002`;
- the Vite origin appears in the API CORS list or falls in the local dev origin rule;
- the browser console is not reporting a CORS, SSL, or generic fetch failure.

For the Docker-assisted path, also check:

- the `retail-api` container is running;
- the project `data/` directory is mounted;
- the mounted DuckDB file exists at `data/retail_revenue_analytics.duckdb`;
- the dashboard image was built with the intended browser-facing API URL.
